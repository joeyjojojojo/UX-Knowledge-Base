
Design-to-Code 设计驱动代码  vs  Prompt-to-Page 意图驱动页面    

Design-to-Code  
典型代表：Figma (Dev Mode)  
UI 精确画稿（**静态矢量图层**）  →  标注/插件解析  →  生成 CSS / 前端框架碎片  
像素级控制 UI 效果，间距、阴影、字体 100% 还原  
能够严格遵循复杂的设计系统 Design System，保持品牌一致性   

代码碎片、冗余，偏向是对UI元素的“视觉标注”，一份“数字说明书”，样式说明书     
产出样式 Styles，而不是完整的逻辑 Logic（只申明按钮是蓝色的，但不知道点击后如何调用SQL数据库）  
仍需要前端工程师把样式组合成一个可工作的组件  


Prompt-to-Page / Generative UI    
典型代表：Google Stitch, V0.dev (Vercel), Claude (Artifacts)  
描述需求/草图（**文本意图、功能逻辑**）  →  AI 编写完整前端代码  →  浏览器直接渲染页面  
AI 生成的是语义化的 HTML/React 结构，自带响应式布局  
功能逻辑完整，包含模拟数据 Mock Data，直接可交互（可运行的组件包）  
UI 样式黑盒，局部细节对话调整效率很低  
风格局限，通常仅有主流框架（如 Tailwind，Material Design）  


起点：静态矢量图层  vs  文本意图、功能逻辑  
产物：样式的说明书  vs  可运行的组件包  
布局逻辑：手动绘制的坐标  vs  自动生成的盒模型（Flex / Grid）  
UX 角色：像素的雕琢者  vs  系统架构的描述者  

翻译  vs  创作  
精准  vs  效率  


Q：从代码还原到 UI，和从 UI 制作成还原度高的代码，中间的 Gap 到底是什么？  
设计与开发断层  

1、使用“代码化”的布局思维 (Auto Layout)   
在 Figma 中，不要使用简单的“组（Group）”，必须全面使用 Auto Layout（自动布局）。----- 关注！  
Auto Layout 的逻辑（Padding, Spacing, Fill/Hug）本质上就是 CSS 的 Flexbox。  
如果你在 Figma 里能用 Auto Layout 实现响应式缩放，前端工程师用一行 display: flex 就能精准还原。如果你的图层是散乱的绝对定位，前端就需要猜你的意图。  

2、建立严格的设计系统 (Design System / Tokens)  
不要在画稿里直接写 颜色: #3B82F6 或 字体: 14px。  
使用设计令牌 (Design Tokens)：定义全局变量，例如 Color-Primary-600 或 Text-Body-M。----- 关注！  
前端同步：确保前端的代码库里有一套一模一样的变量表。当你在 Figma 里修改了 Primary 颜色，前端只需要改一行代码，全局都会同步，避免了手动吸色产生的色差。   

3、遵守“盒子模型”规范 (Box Model)  
设计师需要理解 HTML 的盒子模型  
外边距 (Margin)、内边距 (Padding)、边框 (Border)。  
很多画稿无法还原，是因为设计师在 Figma 里随手拉了一个间距（比如 $13.5px$），而前端规范通常是 $4px$ 或 $8px$ 的倍数。  
建议：开启 Figma 的网格系统（Layout Grid），强迫自己只使用规范的间距数值。----- 关注！  

4、交付完整的“边缘状态” (Edge Cases) ----- 关注！ 
空状态：如果没有数据，页面长什么样？  
极限状态：如果用户名字长达 50 个字符，UI 会不会崩？  
交互状态：悬浮（Hover）、点击（Active）、禁用（Disabled）、加载中（Loading）。  
工具推荐：利用 Figma Dev Mode。它允许开发直接看到间距数值、下载切图，甚至直接拷贝 CSS 变量名。  


Q：Google Stitch 和 Figma Make 如何使用这两个工具快速提高专业技能、建立专业知识体系？  
**“拆解（Deconstruction）”和“逆向工程（Reverse Engineering）”**  

1. 建立“布局逻辑”与“盒子模型”思维   
新人最常见的问题是画出来的 UI 只是“一堆图层”，没有逻辑。  
练习方法：在 Google Stitch 中生成一个复杂的响应式页面（如带侧边栏的数据仪表盘）。  
查看代码结构：观察 AI 是如何使用 Flex 或 Grid 布局的。看它如何定义 Padding（内边距）和 Gap（间距）。  
响应式测试：手动拉伸 Stitch 的预览窗口，观察组件在不同尺寸下是“堆叠”还是“隐藏”。    
目标：将抽象的“视觉间距”转化为具体的“布局协议”，理解为什么 $8px$ 倍数原则在开发中如此重要。    


2. 深度拆解“组件解剖学”（Component Anatomy）  
利用 Figma Make 生成的矢量图层，新人可以学习标准组件的构成。  
练习方法：使用 Figma Make 生成 10 种不同风格的“输入框（Input Field）”或“弹窗（Modal）”。  
图层命名与层级：观察 AI 如何命名图层（如 Label, Hint Text, Container）。  
交互状态定义：查看 AI 是否生成了 Hover（悬浮）、Focus（聚焦）和 Error（报错）等状态。  
属性剥离：选中一个按钮，看它的阴影（Drop Shadow）由几层组成，描边（Stroke）是居中还是内侧。  
目标：建立原子设计（Atomic Design）思维，理解一个合格的组件需要具备哪些物理属性。  


3. 提升“意图转化”与“需求描述”能力  
将模糊的业务需求转化为精准的设计指令。  
正向生成：输入极其简略的 Prompt（如“做一个设置页面”），看 AI 生成的平庸结果。  
迭代描述：逐步增加约束条件（如“包含用户头像、账号安全开关、带有红色警告色值的退出按钮、符合 iOS 规范的间距”）。  
学习重点：对比两次生成的差异。你会发现，你对需求的描述越专业（使用了专业术语，如 Toggle, Avatar, Visual Hierarchy），AI 产出的质量就越高。  
目标：建立专业词汇库，学会用开发和设计通用的语言进行协作。  


4. 建立“多方案评估”的决策体系  
UX 不是为了画出“一个”好看的图，而是为了在多个方案中选择“最优”的一个。  
练习方法：针对同一个功能（如“文件上传进度条”），分别在 Stitch 和 Figma Make 中各生成 5 组方案。  
可用性对比：哪种方案更易读？哪种方案的反馈更明显？  
实现成本对比：哪种方案的代码结构最简单？  
视觉重量对比：哪个方案在整页中太抢戏了？  
目标：摆脱“我觉得这个好看”的感性认知，建立基于逻辑和规范的设计决策体系。  

L1: 结构认知	Stitch	观察代码与布局的对应关系	盒子模型、响应式逻辑、CSS 基础  
L2: 细节拆解	Figma Make	将生成的 UI 转化为组件库	原子设计、图层管理、状态机思维  
L3: 方案决策	两者结合	快速生成多版本进行对比测试	设计评审能力、业务需求转化力
